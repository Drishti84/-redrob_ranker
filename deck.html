Solution Overview
Proposed solution: A two-phase hybrid ranker. Phase 1 (offline) pre-computes sentence embeddings and extracts 30+ structured signals for all 100,000 candidates. Phase 2 (online) embeds the JD locally, computes cosine similarities via a single matrix multiply, scores all candidates with a 5-signal weighted scorer, and produces the ranked top-100 CSV — all in 29 seconds on CPU with zero network calls.
What differentiates this from traditional matching: Traditional systems match keywords — a candidate who lists "FAISS" scores the same whether they used it for 2 months as a beginner or 3 years in production. Our scorer weights each skill by proficiency × duration × endorsements. Career quality distinguishes product-company AI engineers from consulting-only profiles or keyword stuffers (0.40× penalty). A dedicated availability score from 23 behavioral platform signals ensures we surface candidates who will actually respond and join. Honeypot detection removes ~65 impossible profiles. No LLMs involved — every signal is structured and auditable.

JD Understanding & Candidate Evaluation
Key requirements extracted from the JD
Experience: 5–9 years, product company scale, shipped to real users
Vector search: Pinecone, Weaviate, Qdrant, Milvus, OpenSearch, Elasticsearch, FAISS
Embeddings: sentence-transformers, BGE, E5, OpenAI embeddings in production
Evaluation: NDCG, MRR, MAP, A/B testing — not just building, but measuring
LLM fine-tuning: LoRA, QLoRA, PEFT or learning-to-rank (XGBoost, LambdaRank)
Location: India, Pune/Noida, hybrid — availability signals matter
Beyond keyword matching — how we evaluate fit
Skill depth: Each skill hit scored by proficiency (beginner 0.4× → expert 1.0×), duration (0–6 mo: 0.5×, 24+ mo: 1.0×), and endorsement count (+20% max)
Career evidence: ai_role_months measures how long they've actually worked in AI titles — separate from skills they claim to have
Semantic similarity: Cosine distance between the full JD text and the candidate's profile text (all-MiniLM-L6-v2, 384-dim)
Hireability: 23 platform signals — open_to_work, notice_days, response_rate, interview_rate, github_score, days_since_active

Ranking Methodology
Retrieve: All 100K candidates are embedded offline with all-MiniLM-L6-v2 (384-dim, L2-normalised). At rank time the JD is embedded locally → single matrix multiply (100K×384) @ (384×1) gives cosine similarities for all 100K in 0.04 seconds. No FAISS pre-filter — full sweep eliminates recall loss.
Models & algorithms: sentence-transformers for embeddings. Weighted linear combination of 5 hand-crafted, interpretable signal functions. rapidfuzz for fuzzy skill name matching (catches "HuggingFace" vs "Hugging Face Transformers"). No black-box models.

Explainability & Data Validation
How ranking decisions are explained: Every candidate in the top-100 gets a fact-grounded recruiter-style reasoning string generated from extracted structured features — not from an LLM. Example: "Senior AI Engineer at Apple with 5.9 yrs experience; 5+ yrs in AI/ML roles including FAISS, NLP, Python. open to work, 30d notice, high recruiter response rate (80%), India-based, active GitHub (score 97), tier 1 institution." Language is rank-calibrated: ranks 1–10 are assertive, ranks 31–100 include honest concerns (e.g. long notice period, outside India).
No hallucinations or unsupported justifications: Zero LLM calls during ranking. Every fact in the reasoning — title, company, YoE, skill names, availability status — is read directly from the structured feature dict extracted at precompute time. "FAISS" appears in reasoning only if it exists in the candidate's skill list with duration > 0 months. No generation, no inference.
Inconsistent, low-quality, and suspicious profiles: Honeypot detection (5 rules): career date math vs duration_months differs >12 months; expert skill with 0 months used; YoE >5 years beyond total career history; job overlap >180 days; 5+ expert/advanced skills each used <3 months — 65 profiles flagged and excluded. Keyword stuffer penalty: non-AI current title (Marketing Manager, Civil Engineer…) + <12 months AI work history → 0.40× skill score multiplier. Profile completeness and verification status feed directly into availability score.

System Architecture
🗂 precompute.py
Loads candidates.jsonl → features.py extracts title classification, skill hits with proficiency/duration/endorsement weights, ai_role_months, consulting_only, has_product_experience, avg_tenure, education tier, all 23 redrob_signals, full_text for embedding → honeypot.py flags 65 impossible profiles → all-MiniLM-L6-v2 embeds full_text (batch 256, L2-normalised) → saves features.jsonl + embeddings.npy
🎯 rank.py
Loads artifacts (3.7s) → embeds JD from local model cache (10.2s) → dot product similarity all 100K (0.04s) → scorer.py computes 5-component weighted score per candidate, skipping honeypots (1.7s) → sort descending, pick top 100 → reasoning.py generates fact-grounded string per candidate → writes submission.csv
🖥 demo_app/ (Gradio · HuggingFace Spaces)
Loads pre-screened top-1000 features + embeddings (4 MB). User pastes any JD → embed → dot product → score → ranked table with adjustable weight sliders. Runs in ~15s, CPU Basic (free tier).
⚙️ scorer.py — 5 components
skill_match: required_hits coverage × avg weight + text_boost + nice_score. Keyword stuffer penalty: 0.40× if non-AI title + <12mo AI history.

career_quality: starts 0.5 → +0.20 AI title, −0.35 non-AI, +0.20 4+ yrs AI, −0.30 consulting_only, +0.12 production keywords, −0.15 title_chaser.

availability: open_to_work(+0.20), recency, response_rate(×0.18), interview_rate(×0.12), notice_days, location, github_score, verified.

education: tier_1=1.0, tier_2=0.75, tier_3=0.45, tier_4=0.20.
🍯 honeypot.py — 5 detection rules
1. Career date math vs duration_months >12 month gap
2. Expert skill listed with 0 months experience
3. YoE >5 years beyond sum of all role durations
4. Overlapping jobs >180 days
5. 5+ expert/advanced skills each used <3 months

Technologies Used
🧠 sentence-transformers
all-MiniLM-L6-v2 — 22M params, 384-dim. Chosen because it runs on CPU in ~10s for a single JD embedding and produces strong semantic representations for technical English text. No GPU required at inference time.
⚡ NumPy matrix multiply
Replaces FAISS entirely at query time. Embeddings are L2-normalised at precompute → dot product = cosine similarity. (100K×384) @ (384×1) in 0.04s. Eliminates recall loss from approximate nearest-neighbour pre-filtering.
📄 JSONL feature store
Line-delimited JSON vs Python pickle. 100K feature dicts load in 3.7 seconds vs 93 minutes for pickle. Human-readable, no serialisation overhead, portable across Python versions.
🔍 rapidfuzz
Fuzzy partial-ratio matching for skill name normalisation. Catches variations: "HuggingFace" vs "Hugging Face Transformers", "Open Search" vs "OpenSearch", "sentence transformers" vs "Sentence-BERT". Critical for fair skill scoring.
🖥 Gradio + HuggingFace Spaces
Interactive demo — paste any JD, adjust weight sliders, see top candidates ranked in ~15s. Deployed on CPU Basic (free tier). Top-1000 candidate features + embeddings shipped as 4 MB artifacts. No proprietary APIs.
